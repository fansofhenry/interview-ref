Staff member reported Banner showing incorrect award data. "Banner is wrong" can mean 5 things: sync failure, permissions, user error, bad upstream entry, or an actual bug.

Q4

How do you handle a frustrated or upset user? "Describe a time you dealt with a difficult customer..."

Principle: Calm is contagious. The first 15 seconds matter more than the first fix.

Frustration is almost never really about the technology. A faculty member with a broken projector isn't upset about the HDMI cable ‚Äî they're upset because their students are watching them fail in front of 30 people. Respond to the human layer first, then the technical layer.

Two simultaneous tracks:

  • Human track: Acknowledge the situation, give realistic timeline, keep them informed as you work
  • Technical track: Work the problem methodically in parallel
The narration habit: "I'm clearing your browser cache because your session token got stuck ‚Äî if this happens again, Incognito mode will usually bypass it immediately." That sentence explains what I'm doing, gives them a tool for next time, and makes them a participant, not a bystander. Fewer repeat tickets.

Q6

How do you approach training a user or student worker? "How would you mentor a less experienced technician?"

Core concept: The Curse of Knowledge ‚Äî technically skilled people forget what it was like to not understand. Fix that first.

Three principles from my learning science background:

  • Narrate while you show. "I'm right-clicking here because..." builds mental models, not just muscle memory. Muscle memory breaks when the interface changes.
  • Don't solve for the user. Ask "what have you ruled out so far?" before answering. That teaches diagnostic thinking, which is the real skill.
  • Match depth to the person. A faculty member doesn't need to understand session tokens. A student worker troubleshooting SSO issues does. 30 seconds of conversation upfront saves 10 minutes of confusion.
Documentation habit: If I'm showing someone the same process more than once, I write it up as a KB article. Forces clarity. Reduces tribal knowledge risk.
Q7

Tell us about your documentation experience. "How do you approach writing user guides or runbooks?"

Rice University (OpenStax): Operational runbooks for production databases ‚Äî written for the person on call at 2 AM who's stressed, unfamiliar with the system's history, and needs to execute correctly under pressure. Format: unambiguous step sequences, explicit decision points ("if X do Y, if Z escalate to [person]"), no assumed knowledge.

Skyline: Plain-language translations of Oracle error codes for Financial Aid staff. "System error REQD-1044" doesn't help a counselor. "Your account doesn't have permission to view this record ‚Äî submit a ticket with this error code and the student's ID" does.

My standard: Write for the person at their most stressed, not for the person who already knows the answer. Version everything ‚Äî update the doc the day a process changes, archive the old version.
Early priority in this role: Audit the KB for gaps ‚Äî the things people know but haven't written down. Tribal knowledge is a team risk. If it only lives in one person's head, it disappears when they're on vacation.

Q9

Describe a time you identified a systemic issue beyond just solving the immediate problem. "Tell me about a time you found a root cause others had missed..."

The Thursday Afternoon Banner Pattern ‚Äî Skyline

Over 3 weeks, Financial Aid counselors were getting sporadic Banner form errors ‚Äî always on Thursdays, always in the afternoon. Each was resolved individually (clear session, reopen ticket, close). The symptom looked like a session issue, so nobody connected them.

  • Pulled ticket history, sorted by error type and timestamp
  • Every single instance: 4-hour window, Thursday, ~1 PM start
  • That timing = a scheduled process creating database contention
  • Escalated to Banner admin with: "recurring Thursday afternoon session failure correlating with a scheduled job ‚Äî here are the timestamps and affected accounts"
  • Banner admin diagnosed in ~1 hour: a report job misconfigured to run during business hours was degrading active sessions under load
  • Rescheduled to 2 AM ‚Äî Thursday tickets stopped entirely
The habit: A single ticket is a problem to solve. A cluster of similar tickets is a signal. My job isn't just to close the ticket in front of me ‚Äî it's to notice when something is trying to tell me something bigger.

T1

Faculty can't access the Foothill website but can get to Google. DNS vs. server-side failure

  • Ping the Foothill IP directly ‚Äî if it responds, it's DNS; if not, the server itself is down
  • Run tracert foothill.edu ‚Äî see where the connection drops (internal district vs. public ISP)
  • Check browser console for 404/503 ‚Äî 503 = server actively rejecting; 404 = DNS resolves but path is wrong
  • If DNS: flush client cache (ipconfig /flushdns), test with 8.8.8.8 as DNS server
Key phrase: "Google works means your internet connection is fine. Foothill not working means the problem is either DNS resolution or a server issue ‚Äî I narrow it down with ping and tracert before touching anything else."
T2

Projector shows "No Signal" even though laptop is plugged in. Smart classroom multimedia triage

  • Physical chain: HDMI/VGA fully seated both ends + wall plate connection
  • Switcher input: Extron/Crestron panel ‚Äî confirm correct input source selected
  • EDID handshake: Win+P on laptop ‚Üí cycle through Duplicate/Extend to force GPU to re-detect the display
  • If still nothing: try a different cable; test laptop on a different display; check projector power/lamp status
  • Reboot the Crestron/Extron control processor if available
While working: Tell the instructor: "This usually takes 2 minutes. If you want to start class, I can stay quiet in the back." Never make 30 students wait in silence.

Specific time = scheduled process. Don't just reboot ‚Äî find the service.

  • Check Startup tab ‚Äî what's launching at login?
  • Check Windows Event Viewer / scheduled tasks ‚Äî what runs at ~8:45-9:00?
  • Check AD/GPO ‚Äî is there a district-wide AV scan, update push, or login script running then?
  • Check Jamf or SCCM ‚Äî any policy enforcement running at login?
Resolution options: Reschedule the process to off-hours ¬∑ Stagger start times across machines ¬∑ Upgrade RAM if truly insufficient ¬∑ Add exclusion to AV scan for critical work folders
T4

What is the difference between a Permission and a Right in Active Directory? AD conceptual question

Permissions ‚Üí attached to objects (shared folder, printer, mailbox). Control what a user can do TO the object (Read, Write, Full Control).

Rights ‚Üí attached to accounts or groups (via GPO). Control what the user can do ON the system (log in locally, shut down, back up files, manage audit logs).

In practice:

  • Counselor can't open a Banner report folder ‚Üí check Permissions on that network share
  • Staff member can't log into a workstation at all ‚Üí check Rights (local logon right) + AD account status (disabled? locked?)
  • User can log in but can't print ‚Üí check Permissions on the print queue object

T7

Student worker accidentally left another student's Canvas grades visible on a shared lab computer. FERPA incident response

This is a FERPA incident. Treat it that way ‚Äî not as a casual tech issue.
  • Contain immediately: Close the browser session / lock the station ‚Äî stop the exposure first
  • Document facts: Which machine, what time, which student worker, what was visible, whether anyone else saw it
  • Escalate ‚Äî don't handle it yourself: Supervisor + Privacy Officer / Dean of Students. FERPA reporting is a District compliance call, not a desktop technician call.
  • Preventive fix: Configure lab machine auto-lock on idle + review session timeout policy in Jamf or GPO
Principle ‚Äî Privacy by Design: FERPA compliance shapes how I configure systems. Auto-lock, session timeouts, clear-screen policies, and logout training aren't just good practices ‚Äî they're compliance requirements. I bring that lens to every shared-use environment I touch.
T8

Internet "seems slow" in classroom 4210 ‚Äî three other rooms on the same floor too. Network-layer escalation / know your lane

Multiple rooms, same floor = infrastructure issue, not a device problem.
  • Check Freshservice ‚Äî any other tickets from that floor/building this morning?
  • Speed test from affected machine + comparison machine on a different floor
  • Floor-specific + slow on both wired and wireless ‚Üí floor switch or upstream VLAN issue
  • Escalation trigger: Multi-user, floor-specific, not device-level ‚Üí network team with: affected floor, user count, speed test results, start time, wired vs. wireless
  • Post Freshservice status update so additional tickets are linked and users know it's being worked
Know your lane: I can identify which switch or AP serves that floor from network docs, but diagnosis and remediation of switch/AP problems belongs to the network team. I hand off with data, not just a complaint: "Floor 4 seeing 2Mbps down, Floor 3 is fine, started at 9 AM, both wired and wireless affected."
